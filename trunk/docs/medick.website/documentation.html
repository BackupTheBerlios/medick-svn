What is this?

  Medick is a PHP Framework, focused on creating a solid ground for php programmers for developing with fun and ease modern web applications.
  Medick is also a place where I can experiment diffrent programming techniques and where I spend some quality coding time during my weekends.
  Medick is not the place where you should ask questions about how to install php or related software applications. Samples of questions that will remain without an answer: How to build a web form that sends an email?, Help!!!!!, how can I install phpBB, Whats my IP address?, Where can I find a php2exe application or Who shot Kennedy?.
     Medick, the name, came from Black Medick - Medicago lupulina L. Top

Requirements

  

2.1 PHP, version 5.0.3 or higher, 5.1.2 recommended

Extensions and Settings:
  SimpleXML, for parsing app.xml file, it's enabled by default on every php installation otherwise, compile php with: --enable-simplexml
  Reflection, from php 5.1.2 you will also need this extension, for php 5.0.5 this is compiled into the php core and cannot be disabled. Reflection is enabled by default on every php installation, otherwise, compile php with: --enable-reflection
  SPL, enabled by default on every php installation, otherwise, compile php with: --enable-spl
On Windows, this 3 extensions cannot be disabled so you will not have to activate anything to run medick.   

2.2 Web Server, Apache 2.0 recommended

Alltrough I tested medick with LightTPD and Apache 1, Apache 2 with mod_php remains the main development platform.
Apache Setting and modules:
  mod_access should be turned on, and AllowOverride directive from httpd.conf must be set to All - or similar - to allow .htaccess files
  AcceptPathInfo should be On. However, we managed to install a medick application without this directive (can be triky with php as CGI) and I'm planning to remove this requirement from medick 0.2.
  mod_rewrite for nice URLs.   

2.3 Database Server

If you are going to use a database as a backend for your application, medick commes with Creole bundled.
So, medick will work with whatever sql driver is supported by Creole and by your php installation. This includes: mysql, pgsql, sqlite, mssql, oracle, odbc or mysqli.

Getting Started :: elTodo on your computer

  The online elTodo sample is located here. This is what I will try in this section to install on your computer.
  Grab yourself a copy of medick, either download an almost stable release (see our front page), or do a checkout from the project subversion tree. In this short section, we will go with the second option.
  First, make sure you have Subversion, the client libraries, installed on your computer. For Windows a client like TurtoiseSvn will do the job.
  Also, check again if you meet medick requirements and that you have all the required software installed and properly working.   

4.1 Checkout the Sources

$ svn checkout svn://svn.berlios.de/medick medick
This will create a medick folder under the current working directory, we will refer to this as $MEDICK_HOME from now on. Subfolders will include trunk/, branches/ and applications/.   

4.2 Symlink or Make the Public, PUBLIC

Next step is to copy or to create a symbolic link under linux from the $MEDICK_HOME/applications/eltodo to your web server document root. Change the working directory to document root, and do:
$ ln -s $MEDICK_HOME/applications/eltodo/public eltodo
Under Windows, just move $MEDICK_HOME/applications/eltodo/public to eltodo under your web server document root. Same for linux if the above command will not do.   

4.3 Database Setup

elTodo application can use mysql or sqlite databases as backend. We will use mysql since everyone is familiar with it.
Import $MEDICK_HOME/applications/eltodo/db/schema.mysql database schema:
$ mysql -u $USERNAME -p$PASSWORD < $MEDICK_HOME/applications/eltodo/db/schema.mysql
Replace $USERNAME and $PASSWORD with your real database username/password. This should also work under Windows. If not, you can use phpMyAdmin or other database clients.   

4.4 Final Touch

This section will be deprecated for medick 0.2.0, since many of those settings will go to only one file, the application_name.xml.
Create an empty file, eltodo.log on $MEDICK_HOME/applications/eltodo/log/eltodo.log for logging. 
$ touch $MEDICK_HOME/applications/eltodo/log/eltodo.log
$ chmod 777 $MEDICK_HOME/applications/eltodo/log/eltodo.log
Open $MEDICK_HOME/applications/eltodo/conf/eltodo.xml and configure the application to match your setup.
For sure you will need to change under Logger System section, the property named path, to use yours.
Also, under Database Settings you will need to change the user and pass sections.
Make sure that server_name and document_root will also work for you.
Open $MEDICK_HOME/applications/eltodo/public/.htaccess and change the line with MEDICK_APPLICATION_PATH to point to $MEDICK_HOME/applications/eltodo
Open $MEDICK_HOME/applications/eltodo/public/index.php and change the include_once to $MEDICK_HOME/trunk/boot.php.
Uff, so, are you still with me? Great. So who said "convention over configuration"?
Well, we almost did it!. Just point your browser to http://localhost/eltodo and...   

4.5 What Went Wrong?

For php installed as a CGI, I had to manually define MEDICK_APPLICATION_PATH and MEDICK_APPLICATION_NAME: 
// in index.php before include_once
$_SERVER['MEDICK_APPLICATION_PATH']='$MEDICK_HOME/applications/eltodo';
$_SERVER['MEDICK_APPLICATION_NAME']='eltodo';
Remember to replace $MEDICK_HOME with your actual path!
If you get a blank page, try to turn on display_errors setting from php.ini.
You get a 403 - Forbidden page, and on apache logs you can read: 
[error] [client 127.0.0.1] Symbolic link not allowed: [....]
This usually means that the user that runs apache is not allowed to read from $MEDICK_HOME/applications/eltodo/public as a quick solution, try to create a VirtualHost for elTodo application or try to move $MEDICK_HOME under ServerRoot. However, I was unable to reproduce this problem under Windows.
My setup includes ServerRoot to /wwwroot, DocumentRoot on /wwwroot/htdocs and $MEDICK_HOME to /wwwroot/medick and it just works.
Please contact us if you cannot setup the elTodo sample.
More to be added as we discover more problems...   

4.6 Using the Downloadable Version

If you choose to download medick, just unpack the tgz archive somewhere: 
$ tar -xzvf medick-x.y.z.tgx
For Windows you can use 7-Zip to unpack.
The resulting folder will be your $MEDICK_HOME. But instead of trunk/ you will have framework/ and applications/ is replaced by samples/. The above steps will also work for this type of setup.

5.4 Controllers, Actions and URLs

In medick framework, Controller classes are made to handle requests made by the user. Any incoming URL is mapped to a Controller class and to a method within that class.
If you type an URL, like http://localhost/hello/foo you will get an "Server Error" with the message "Cannot Process your request!" and as details "RoutingException - Message: Cannot find a Route with this name: 512". Also a nice backtrace is shown there.

Click on image for a larger view.

Now we can use the other medick helper script, witch can generate for us a new Controller. It's located on the application folder/scripts, hello/scripts/generator.php. The first parameter that this script takes is controller (what we want to generate, later on we will use the same script for creating other helper scripts), and the second is the name of that controller, basically the controller class name. I choose to use main as the name, and from $MEDICK_HOME/applications, I ran:
[aurelian@gremlin applications]$ php hello/scripts/generator.php controller main
Creating controller class MainController from main
    create /wwwroot/medick/applications/hello/app/views/main
    create /wwwroot/medick/applications/hello/app/views/layouts/main.phtml
    create /wwwroot/medick/applications/hello/app/helpers/main_helper.php
    create /wwwroot/medick/applications/hello/app/controllers/main_controller.php
    
Medick ($v: 0.2.0-svn) [ DONE ].
As you can see, the generator just created for us a Controller class MainController located on the file $MEDICK_HOME/applications/hello/app/controllers/main_controller.php.

Click on image for a larger view.

If you load the page http://localhost/hello/main you will get another error, but this means that we are on the right trak, an action named "index" dosen't exists for the route "main" - or in php therms, the class MainController dosen't have a method named "index".
We will use the generator again to create that method. We will run the same command but this time we add a new parameter witch is the action method name that we want to create. Infact, everything after the controller name is interpreted as an action to be added on that controller.
Ok, so, lets create 2 methods, index and say for the main controller:

Project History

  The idea of writing my own php framework raised after I've read the article "An introduction to the Model-View-Controller Pattern" written by Jason Sweat in may 2003 issue of php|architect.

  The first working "version" was used, somewhere in August 2004, for an internal web site for the company that I was working for.
  This "core" engine, was integrating it self very well with Smarty and AdoDB, witch, at that time represented the M an V from MVC. Moreover, a caching system was wrote since it was an web site about large statistics witch required a lot of data manipulation.

  Next projects where I was involved as a freelancer, reuse almost the same ideas and core engine.
  Unfortunately, from that period of time, http://www.roumanie-france.ro/ is the only web site that is still on line
Again Smarty and Ado DB, but also a modified version of PEAR_DB_Nested_Set (you can see it on the left menu and on the web pages breadcrumbs) and the excellent phpOpenTraker for web statistics. The same caching engine is still used.
  On the administrative side, the user can perform various actions with the web pages (add/edit/move to other parents or delete/de-activate).

  In April-May 2005 I was starting to think about registering this ideas as an open-source project, and in June 2005, medick was born (it didn't have a name until then) on http://developer.berlios.de.
  On the same period, the ruby web framework, RubyOnRails, started to raise the hype. I have enjoyed playing with many of the RoR features and I wanted to have something similar for medick.
